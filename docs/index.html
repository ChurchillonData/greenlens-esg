Corporate sustainability reports are structurally difficult to audit. Claims range from specific quantified commitments ("reduced Scope 1 emissions by 37% against our 2019 baseline") to vague aspiration ("we support the energy transition"). Claimify is a RAG (Retrieval-Augmented Generation) pipeline that scores each claim against a curated NGO evidence corpus rather than relying on the LLM's training weights alone. Retrieval gives the model citable, up-to-date source material; generation produces a structured verdict grounded in that material, not in generalities about the sector.

The pipeline runs entirely offline. No API calls happen at read time. The frontend reads two pre-generated JSON files served as static files on Vercel.

Pipeline stages
PDF download→ pdfminer parse→ ClimateBERT filter→ GPT-4o extract→ SBERT + rerank→ GPT-4o-mini score→ Materiality weight→ React / Vercel
01

Background and Design Decisions

Oil and gas sustainability reports are written for multiple audiences at once: shareholder, regulator, and general public. The same company can, on the same page, disclose a precise Scope 1 reduction figure and an unfalsifiable aspiration in the next paragraph. Distinguishing between them is not a keyword problem or a sentiment problem. It requires knowing whether a claim makes a falsifiable factual assertion and, separately, whether available evidence refutes it.

Existing approaches had a shared gap: they either flagged anything containing "net-zero" (keyword matching) or relied on journalists manually cross-referencing company statements with NGO reports. Neither scales, and neither produces a traceable reasoning chain. This pipeline is the automation of that intermediate layer.

The choice to use RAG rather than a prompt-only classifier is deliberate. LLM training weights are frozen at a cutoff date and do not contain the specific Carbon Tracker reports or InfluenceMap briefings that contradict individual company claims. Giving the model that material at inference time means every verdict traces back to a specific retrieved document.

02

System Architecture

Two input streams meet at the retrieval stage. The claims stream processes sustainability PDFs through the NLP pipeline. The evidence stream ingests NGO sources and builds a vector index. Both feed the RAG retrieval and scoring pipeline. A separate Commitment Tracker branch (dashed) reuses the same retrieval infrastructure with 2021 historical pledges as its input.

03

NLP Pipeline: Filtering and Extraction

Before retrieval can happen, the pipeline needs claims. A 300-page corporate sustainability report typically contains thousands of sentences. Most are financial tables, legal disclaimers, and general corporate narrative. Running GPT-4o over all of them would be slow, expensive, and noisy. Three steps handle this: sentence splitting, ClimateBERT relevance filtering, and structured claim extraction.

ClimateBERT is the gating step. It runs locally with no API call and processes sentences in batches of 32. In practice it drops 60–70% of a typical report before GPT-4o sees anything.

ClimateBERT relevance filter

nlp/relevance_filter.py
MODEL_NAME = (
    "climatebert/"
    "distilroberta-base"
    "-climate-detector"
)
BATCH_SIZE = 32

def predict_batch(
    sentences: list[str]
) -> list[bool]:
    tokenizer, model = get_model()
    inputs = tokenizer(
        sentences,
        padding=True,
        truncation=True,
        max_length=256,
        return_tensors="pt"
    )
    with torch.no_grad():
        outputs = model(**inputs)
    preds = torch.argmax(
        outputs.logits, dim=-1
    ).tolist()
    # label 1 = climate-relevant
    return [p == 1 for p in preds]

Why ClimateBERT and not another GPT-4o call? An early version asked GPT-4o to decide relevance before extraction. Accurate but doubled the API cost for a task ClimateBERT handles locally in seconds. A missed sentence means one fewer candidate claim, not a false positive in the final output. Precision matters more than recall at the filter stage.

04

Evidence Retrieval

This is the R in RAG. For every claim, the pipeline retrieves the most relevant passages from the NGO evidence corpus before the LLM ever sees it. The model is grounded in specific documents, not in training-weight generalities about oil companies and climate.

Retrieval uses two stages. SBERT computes cosine similarity across the corpus and returns a top-20 shortlist filtered to the claim's company. The cross-encoder then re-scores every (claim, candidate) pair with full cross-attention — significantly more accurate but too slow to run against the full corpus. Top-5 go to the scorer.

retrieval/sbert_search.py
def cosine_scores(
    claim_vector: np.ndarray,
    corpus_vectors: np.ndarray
) -> np.ndarray:
    # L2-normalise → dot = cosine sim
    cn = claim_vector / np.linalg.norm(
        claim_vector
    )
    vn = corpus_vectors / np.linalg.norm(
        corpus_vectors,
        axis=1, keepdims=True
    )
    return vn @ cn

def search(
    claim_vector, company_id, k=20
):
    records, vectors = load_index()
    scores = cosine_scores(
        claim_vector, vectors
    )
    r, s = filter_by_company(
        records, scores, company_id
    )
    return top_k(r, s, k)
retrieval/rerank.py
MODEL = (
    "cross-encoder/"
    "ms-marco-MiniLM-L-6-v2"
)

def rerank(
    claim_text: str,
    candidates: list,
    top_n: int = 5
) -> list:
    pairs = [
        (claim_text, rec["text"][:512])
        for rec, _ in candidates
    ]
    scores = get_model().predict(pairs)
    paired = list(zip(
        [r for r, _ in candidates],
        scores
    ))
    paired.sort(key=lambda x: -x[1])
    return paired[:top_n]

Why PDF chunking mattered: Early corpus versions stored whole PDFs as single records. SBERT's 512-token limit meant the model was scoring a truncated title-page fragment. 05.A

What Makes This RAG Non-Standard

The core mechanics are off the shelf: SBERT embeds, cosine ANN shortlists, cross-encoder reranks, GPT-4o-mini generates. The differentiation is in four design decisions layered on top of the retrieval stack, each motivated by a specific failure we observed.

Generic RAGClaimify RAGWhy the change was necessary
Retrieve from full corpus Company-scoped retrieval Without it, a BP claim retrieved Shell methane documents and scored incorrectly. The applies_to field narrows the candidate pool before SBERT runs. Cross-company contamination is invisible until you inspect individual retrieved passages.
Single LLM call: classify and cite Two-step classification Single-step prompts inflated contradicted verdicts — the model treated sector-wide NGO criticism as company-specific falsity. Splitting into two questions dropped the false-contradicted rate and improved accuracy by ~12 percentage points.
All documents treated equally Credibility-weighted corpus A Guardian article and a Carbon Tracker financial model should not score equally. Credibility weights (0.75–0.95) scale the cross-encoder score before top-5 selection, promoting higher-quality sources without ignoring relevance.
Single retrieval loop per query Second-order temporal loop The Commitment Tracker runs the same retrieval stack with a 2021 pledge as input and 2023–2025 NGO evidence as the judge. Same mechanics, different epistemology: not "is this supported?" but "was this kept?"

What this pipeline does not do: No fine-tuning on climate domain text. Retrieval runs in-memory with numpy, not a vector database. No HyDE, no self-querying, no iterative retrieval loop. These would likely improve quality but increase complexity and inference cost. The architecture was sized to run offline on a single machine within a predictable API budget.

Where the accuracy gains actually came from

Two changes drove almost all of the improvement from 78.3% to 86.7%:

Two-step classification rule

The largest jump was 73.3% → 83.3% when the prompt was restructured to separate the well/weakly decision from the contradicted check. The model's tendency to conflate "NGO criticises company" with "claim is false" was the dominant error mode, and separating the two questions removed it.

PDF chunking (350-word / 50-word overlap)

The second jump was 83.3% → 88.3% after chunking. Whole-PDF records truncated at SBERT's 512-token limit — the model embedded a title-page fragment, not technical content. Scope 3 claims improved most because relevant evidence appears in appendices.

Company-scoped retrieval and credibility weights are correctness guardrails, not accuracy drivers. The corpus weighting has not been formally ablated; its contribution may be smaller than expected.

06

Claim Classification and GreenSignal Score

Once retrieval selects the top-5 evidence passages, the claim and passages are passed to gpt-4o-mini in a single prompt. The model returns a class, a risk score, and a reasoning string. A second call generates the user-facing rationale grounded in the same evidence.

scoring/llm_scorer.py
def build_user_prompt(
    claim_text: str,
    evidence: list[dict]
) -> str:
    # This is the RAG injection point.
    # Claim + retrieved docs in one prompt.
    ev_text = "\n".join(
        f"[{i+1}] {e['text'][:400]}"
        for i, e in enumerate(evidence)
    )
    return (
        f"CLAIM:\n{claim_text}\n\n"
        f"RETRIEVED EVIDENCE:\n{ev_text}"
    )

Two-step decision rule

Step 1 — claim text only

Does the claim contain at least one concrete element? A company-owned quantity, a named deadline, or a completed named deal. If yes: well_substantiated. If no: weakly_substantiated. Retrieved evidence plays no role in this step.

Step 2 — evidence check

Does any retrieved passage name the company and the specific commitment and show it is false or walked back? If yes: contradicted. Sector-wide criticism does not qualify. The bar is high by design.

GreenSignal: materiality-adjusted final score

07

Commitment Tracker

The Commitment Tracker answers a question the main pipeline does not: of the climate pledges companies made in 2021, how many have they actually kept? It reuses the same retrieval stack but runs it in reverse time — 2021 sustainability reports as input, 2023–2025 NGO evidence corpus as the judge.

The pipeline extracts 710 historical pledges, embeds them with the same SBERT model, retrieves the most relevant current evidence for each, and asks GPT-4o-mini to issue a gap verdict. The retrieval mechanics are identical to the main pipeline; the epistemology is different.

Accountability score formula

scripts/gen_tracker_data.py
VERDICT_PENALTY = {
    "reversed":            1.0,
    "partially_fulfilled": 0.5,
    "fulfilled":           0.0,
}

def accountability_score(scored):
    # 0 = all reversed
    # 1 = all fulfilled
    total_w = penalty_w = 0.0
    for r in scored:
        w  = CATEGORY_WEIGHT.get(
               r["category"], 1.0)
        wc = w * (r["confidence"] or 0.5)
        total_w   += wc
        penalty_w += (
            VERDICT_PENALTY[r["verdict"]]
            * wc
        )
    return round(1.0 - penalty_w/total_w, 3)

08

Evaluation

The eval set is 60 claims hand-labelled by the author using the same rubric encoded in the system prompt. This tests whether the model applies the rubric consistently — not whether the rubric itself is correct. The non-monotonic accuracy path (78% → 71% → 88%) is informative about which changes helped and which caused regressions.
